# 从PDF教材里自动提取学习路径：踩了5个坑才搞定的全记录

> 记录一个传播学教材学习路径生成器的开发过程——停用词bug、中文分词陷阱、n-gram噪声过滤……每一步都是真实踩过的坑。

## 写在前面

我在学习传播学教材时，面对24页的"第四章：符号互动"，感到内容庞杂，不知道从何学起。

我想：能不能写个程序，自动从PDF中提取核心概念，分析它们之间的前置关系，然后生成一个合理的学习顺序？

于是有了这个项目。本文记录了开发过程中遇到的5个主要问题，以及每一次是如何发现、诊断和解决的。最终版本能从同一章节中提取**19个核心概念**（初始版本仅7个），并生成包含时间估算和难度分级的**5步学习路径**。

这不是什么高深论文，就是一份踩坑备忘录。希望对你有用。

**开源地址**：https://github.com/back1992/study-route

## 系统架构（简单说一下）

系统分两个阶段：

**阶段1：知识提取（knowledge-base包）**

- 输入：PDF文件
- 用PyMuPDF解析PDF，合并断行
- 多策略提取概念和定义
- 输出：概念列表、定义、章节摘要

**阶段2：路径规划（study-route包）**

- 输入：概念列表
- 用NetworkX构建前置关系图
- Kahn算法做拓扑排序和分层
- 输出：学习步骤序列

前置关系推断有两种方式：算法推断（基于图结构）+ 可选的LLM增强。

技术栈：Python 3、PyMuPDF、NetworkX、可选OpenAI API。

## 坑1："装车过程"是怎么混进来的？

### 问题

提取结果中出现了"装车过程"，定义为"把信息加载到符号上面，而符号就等同于传播信息的运输工具"。

这显然不是学术概念。翻开原文，这是作者解释"编码"时用的比喻：

> "不妨设想有件集装箱，从郑州运往广州……这个所谓装车过程就是把信息加载到符号上面……"

### 诊断

我检查了停用词列表——"装车"和"过程"都在里面！为什么没被过滤掉？

问题出在过滤逻辑：

```python
def is_valid_term(text):
    if text in STOPWORDS:  # 精确匹配！
        return False
    return True
```

**这是精确匹配，不是子串匹配。** "装车过程"既不等于"装车"，也不等于"过程"，所以通过了过滤。

### 修复

改成子串匹配：

```python
def is_valid_term(text):
    for stopword in STOPWORDS:
        if stopword in text:  # 子串匹配
            return False
    return True
```

### 反思

典型的"想当然"错误。写代码时心里想的是"包含停用词就过滤"，手上写的是"等于停用词就过滤"。测试时用的都是2-3字短词，没暴露问题。

## 坑2：为什么核心概念提取不到？

### 问题

修好停用词bug后，"装车过程"消失了，但核心概念缺失的问题还在：

- **"译码"**：整章反复出现，但没提取到
- **"选择性定律"**：第三节标题，没提取到
- **"经验范围"**、**"劝服"**、**"创新扩散"**：全都没提取到

### 诊断

逐条检查这些概念在原文中的出现方式：

**"译码"**——句子是"译码同编码正好相反"、"说话是编码，听话就是译码"。没有"是指"、"定义为"这样的定义句式，正则匹配不到。

**"选择性定律"**——只出现在章节标题"第三节 选择性定律"中，正文没有定义句。

**"经验范围"**——虽然有"经验范围是指……"这样的句子，但因为出现次数少，被后续过滤逻辑丢掉了。

**根本原因**：正则模式匹配只能捕捉有明确定义句式的概念。对于那些反复出现但没有"X是指Y"句式的概念、出现在章节标题中的主题概念，正则方法无能为力。

### 修复：从章节标题提取

既然章节标题本身就是最好的概念来源，那就加一个策略：

```python
def extract_section_titles(text):
    # 匹配 "第一节 编码与译码"
    pattern = r'第[一二三四五六七八九十]+[节章]\s+([\u4e00-\u9fff\s与和]+)'
    
    titles = []
    for match in re.finditer(pattern, text):
        title = match.group(1).strip()
        # 拆分："编码与译码" → ["编码", "译码"]
        parts = split_by_conjunctions(title, ['与', '和', '及'])
        titles.extend(parts)
    
    return filter_generic_words(titles)
```

提取了"符号互动"、"译码"、"选择性定律"、"人际传播"、"人际影响"、"媒介效应"。

**新问题**：也提取了"左右认知"、"心理机制"这些明显不是核心概念的词汇。

### 补丁：手工维护通用词列表

```python
GENERIC_WORDS = {
    '左右认知', '心理机制', '编码活动',
    '认知', '感知', '机制', '心理', '活动',
    ...
}
```

看到什么加什么。不够优雅，但能用。

## 坑3：如何捕捉高频但无定义的概念？

### 问题

即使加上了章节标题提取，"经验范围"、"舆论领袖"、"两级传播"还是没提取到。它们在正文中反复出现（经验范围出现了27次！），但没有标准的定义句式。

### 思路：CJK字符n-gram分析

中文没有空格分隔，不能像英文那样用split()分词。我又不想引入jieba（增加依赖）。

折中方案：提取所有2-5字符的连续中文序列，统计它们在多少个段落中出现。

```python
def extract_cjk_ngrams(text, min_paragraphs=5):
    paragraphs = re.split(r'\n\n+|[。！？]', text)
    ngram_counts = {}
    
    for para in paragraphs:
        for n in [2, 3, 4, 5]:
            for i in range(len(para) - n + 1):
                ngram = para[i:i+n]
                if is_pure_cjk(ngram):
                    ngram_counts[ngram] += 1
    
    candidates = [(ng, cnt) for ng, cnt in ngram_counts.items() 
                  if cnt >= min_paragraphs]
    return sorted(candidates, key=lambda x: -x[1])
```

### 运行结果：一堆噪声

Top 10长这样：

- '传播' (108次)
- '信息' (98次)
- '理解' (63次)
- '人们' (55次)
- '事物' (47次)
- '一个' (43次)
- '就是' (41次)

问题很明显：2字符通用词因为出现频率高，占据了前列。真正的领域概念被挤到了后面。

### 修复过程（连续4个补丁）

**补丁1：优先长n-gram**

把排序改成"按长度降序，再按频率降序"：

```python
candidates.sort(key=lambda x: (-len(x[0]), -x[1]))
```

4-5字符的术语会排在前面了。

**补丁2：过滤函数词**

"的"、"了"、"在"虽然是CJK字符，但是函数词：

```python
FUNC_CHARS = set('的了是在把被从与和及')
if ngram[0] in FUNC_CHARS or ngram[-1] in FUNC_CHARS:
    continue
```

**补丁3：子串去重**

"选择性接"是"选择性接触"的子串，只保留后者。

**补丁4：排除已提取的概念**

"译码"已经通过章节标题提取过了，不需要重复。

### 一个让我困惑的bug

我把频率分析加了一个条件：

```python
if len(terms) < 10:
    # 概念不足10个时才运行
    run_frequency_analysis()
```

结果正则+章节标题已经提取了15个概念，条件为假，频率分析**根本没运行**。

改成 `if True:` 后，又发现频率分析只添加了1个概念。加debug打印后发现：运行前 `len(terms)` 已经是21了（之前步骤有重复项），触发了 `>= 20` 的提前退出。

修复：在频率分析前先做一次去重。

## 坑4：CJK字符的陷阱

### 问题

扩展到5字符n-gram后，结果中出现了奇怪的术语：

- **'的经验范围'** (15次) ← 多了个"的"
- **'编码与译码'** (13次) ← 这是章节标题
- **'传播学引论'** (11次) ← 这是书名！
- **'播学引论（'** (11次) ← 带了个左括号

### 诊断

原文中有"……的经验范围的大小会影响传播效果……"，n-gram从"的"开始截取5个字符，得到了"的经验范围"。

"传播学引论"出现在每页页眉"传播学引论（第四版）"，频率很高。

### 修复

三个修复叠加：

**修复1**：过滤以函数词开头的n-gram（之前只检查了开头字符是否CJK，没检查是否是函数词）

**修复2**：把"传播学引论"等加入噪声黑名单

**修复3**：要求n-gram必须全部是CJK字符，过滤掉带标点、数字的

```python
if not all('\u4e00' <= c <= '\u9fff' for c in ngram):
    continue
```

## 坑5：部分匹配 vs 完整术语

### 问题

提取结果中出现了：

- '选择性接触' ✅ 完整术语
- '选择性接' ❌ 不完整
- '择性接触' ❌ 不完整
- '选择性理解' ✅ 完整术语
- '选择性理' ❌ 不完整

"选择性接"、"择性接触"这些都是完整术语的"切片"，应该被过滤。

### 诊断

n-gram分析提取了所有4-5字符的序列。"选择性接触"是5字符，"选择性接"是4字符。

之前的子串去重只检查"n-gram是否是另一个更长n-gram的子串"，但"选择性接"不是"选择性接触"的子串（长度差1，位置不同）。

### 修复

检查n-gram是否是某个更长n-gram的前缀或后缀：

```python
def remove_partial_matches(candidates):
    candidates.sort(key=lambda x: -len(x[0]))
    
    result = []
    for ngram, count in candidates:
        is_partial = False
        for longer, _ in result:
            if len(longer) > len(ngram):
                if longer.startswith(ngram) or longer.endswith(ngram):
                    is_partial = True
                    break
        if not is_partial:
            result.append((ngram, count))
    return result
```

这个方案有点粗糙——更严谨的做法是检查n-gram在原文中的边界（前后是否有空格或标点），但中文没有空格，边界判断很复杂。目前"能用就行"。

## 学习路径是怎么生成的

概念提取完成后，还有三步：

### 第一步：推断前置关系

两种方法结合：

**算法推断**：如果上游提供了概念关系图，根据关系类型（prerequisite、related_to等）直接转换。

**LLM增强（可选）**：把概念列表发给LLM，让它推断哪些概念需要先学哪些。

```
Prompt大意：
"以下是概念列表：[编码, 译码, 选择性定律, ...]
请分析哪些概念是其他概念的前置知识？
输出JSON格式"
```

合并两者结果，去重。

### 第二步：构建DAG + 拓扑排序

用NetworkX构建有向图，Kahn算法做分层拓扑排序。

**分层的意义**：同一层内的概念没有前置依赖，可以并行学习。不同层之间有严格先后顺序。

```
第1层: ['编码', '译码']      ← 先学这两个
第2层: ['符号互动']          ← 需要先理解编码和译码
第3层: ['选择性定律']        ← 需要先理解符号互动
...
```

如果LLM推断出了循环依赖（A→B→C→A），就检测并打破环——粗暴地删掉环的最后一条边。

### 第三步：组装学习步骤

每一层转换成一个学习步骤，加上时间估算和难度分级：

- **时间**：每个概念5分钟（简单粗暴但能用）
- **难度**：前30%的步骤标"easy"，中间40%标"medium"，后30%标"hard"

## 经验总结

### 关于调试

**加debug打印是最快的诊断方法。** 在关键位置打印中间结果，比单步调试高效得多。

**问题往往出在边界条件。** 精确匹配vs子串匹配、n-gram的边界判断、术语数量阈值……写代码时容易忽略，运行时就暴露。

**中文处理有很多特殊情况。** 没有空格分隔、函数词也是CJK字符、标点混在文本中……英文处理中不存在的坑，中文处理必须逐一解决。

### 关于架构

**多策略融合是必要的。** 正则适合有定义句式的概念，章节标题适合主题概念，频率分析适合高频但无定义的概念。单一策略无法覆盖所有情况。

**去重逻辑比提取逻辑更复杂。** 三种策略各自提取概念，合并时要去重、去子串、去噪声。这部分代码占了整个提取流程的一半。

**LLM是增强而非替代。** LLM可以推断前置关系，但概念提取还是得靠规则。LLM输出不稳定，完全依赖它会导致结果不可复现。

### 关于工程实践

**"能用就行"的实用主义。** 停用词列表手工维护，难度估算简单线性，时间估算固定5分钟/概念。不够优雅，但能解决问题。

**迭代开发比一次性设计更高效。** 第一版用了1小时，问题很多。之后每发现一个问题就修复一个，逐步迭代到最终版本。如果一开始就设计"完美"方案，可能到现在还没写完。

**测试数据要真实。** 人工构造的测试数据上，第一版就能跑通。真实PDF一上，各种问题都暴露了。尽早用真实数据测试。
